**版本**: v1.0

**状态**: 设计批准，待实施

## 项目概述

ScreenTrace 是一个个人时间管理工具，通过定期截屏和多模态大模型API分析，自动记录用户一天的活动，并生成周报和统计报告。

**核心价值**：
- 自动化时间追踪，减少手动记录负担
- 多维度分析工作/学习/休闲时间分配
- 生成详细周报，帮助优化时间管理

**目标用户**：需要时间管理和工作效率优化的个人用户

## 核心需求

### 功能需求
1. **智能截图采集**
   - 固定时间间隔截图（可配置，默认10分钟）
   - 窗口切换时触发截图
   - 智能去重：距离上次截图<2分钟或图像相似度>85%时跳过

2. **隐私保护**
   - 应用黑名单（如密码管理器）
   - 窗口标题关键词黑名单（如"密码"、"银行"）
   - 内容关键词黑名单（可选OCR识别）
   - 匹配时跳过截图并记录日志

3. **多模态AI分析**
   - 支持多种API（OpenAI/Claude/Gemini）
   - 以OpenAI规范为主，用户可配置切换
   - 提取信息：应用名称、生活维度、活动形式、详细描述、关键词

4. **数据管理**
   - SQLite数据库存储元数据
   - 截图按日期分文件夹存储
   - 自动清理旧数据（可配置保留天数）
   - 未来支持加密（v1.1）

5. **多维度报告生成**
   - 时间线日志：按时间顺序列出任务
   - 任务归类统计：按时长分类汇总
   - 叙述式报告：日记风格的总结
   - 可视化图表：饼图、柱状图、时间轴

6. **统计面板**
   - Web页面展示（localhost HTTP服务）
   - 交互式图表（Plotly/Echarts）
   - 时间范围筛选（日/周/月）
   - 应用使用统计、趋势分析

### 非功能需求
- **性能**：静默运行，CPU占用<5%，内存<200MB
- **可用性**：7x24小时稳定运行
- **隐私**：本地存储，API调用仅发送截图和分析请求
- **可扩展性**：模块化设计，便于未来迁移到Electron

## 技术架构

### 技术选型

**方案：Python + 系统调用**

**理由**：
- Python生态成熟，截图、窗口检测库丰富
- 开发速度快，适合快速验证MVP
- 未来可迁移到Electron（数据格式通用）
- 打包体积适中（<50MB）

```
- 截图：Pillow / pyautogui

- 数据库：sqlite3
- API调用：requests / httpx

- 打包：PyInstaller
```

## 核心组件设计

### 1. 监控调度器（Monitor Scheduler）

**职责**：协调截图触发时机

**子组件**：
- **定时器线程**：每N分钟检查是否需要截图
- **窗口监听器**：监听Windows钩子事件，检测窗口切换
- **去重决策器**：根据时间间隔（<2分钟）和图像相似度决定是否跳过

**实现要点**：
- 使用 `threading.Timer` 实现定时器
- 使用 `win32gui.SetWinEventHook` 监听窗口切换
- 线程安全设计，避免竞态条件

### 2. 黑名单过滤器（Privacy Filter）

**职责**：保护隐私，避免敏感信息泄露

**过滤流程**：
```
1. 应用黑名单检查 → 检测当前窗口进程名
2. 标题黑名单检查 → 窗口标题关键词匹配
3. 内容黑名单检查 → OCR识别（可选，默认关闭）
4. 任一命中 → 跳过截图，记录日志
```

**配置示例**：
```json
{
  "blacklist_apps": ["1Password.exe", "Bitwarden.exe"],
  "blacklist_title_keywords": ["密码", "银行", "支付"],
  "blacklist_content_keywords": ["password", "secret"]
}
```

### 3. 多模态API调用器（Vision API Client）

**职责**：调用大模型API分析截图内容

**核心功能**：
- 配置管理：支持OpenAI/Claude/Gemini等API
- 图像预处理：压缩至合适分辨率（节省token）
- Prompt模板：标准化询问
- 错误重试：失败时指数退避重试（最多3次）
- 成本控制：记录API调用次数和token消耗

**Prompt模板设计**：
```
你是一个活动记录助手。请分析这张屏幕截图，提取以下信息：

1. 【应用名称】：当前使用的应用或网站

2. 【生活维度】：从以下分类中选择最匹配的
   - 工作（职业相关活动）
   - 学习（自我提升、教育相关）
   - 休闲（娱乐、放松）
   - 生活（日常事务）
   - 其他（请注明）

3. 【活动形式】：从以下分类中选择
   - 创作/操作（编写、设计、操作软件）
   - 阅读/观看（文档、视频、文章）
   - 沟通协作（会议、聊天、邮件）
   - 浏览检索（搜索、浏览网页）
   - 其他（请注明）

4. 【详细描述】：描述具体在做什么（包含内容主题+具体活动）
   格式：<活动动词> + <内容主题> + <具体细节>
   示例：
   - "观看 Python 多线程教程视频"
   - "阅读《海贼王》漫画"
   - "编写用户认证模块代码"

5. 【关键词】：提取3-5个关键词

请以JSON格式输出：
{
  "app": "应用名称",
  "life_category": "生活维度",
  "activity_form": "活动形式",
  "description": "详细描述",
  "keywords": ["关键词1", "关键词2"]

```

### 4. 数据管理器（Data Manager）

**职责**：管理数据存储和生命周期

**数据结构**：

**数据库表设计**：
```sql
CREATE TABLE screenshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp DATETIME NOT NULL,
    app_name TEXT,
    life_category TEXT,
    activity_form TEXT,
    description TEXT,
    keywords TEXT,  -- JSON数组
    screenshot_path TEXT NOT NULL,
    similarity_score REAL,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_timestamp ON screenshots(timestamp);
CREATE INDEX idx_life_category ON screenshots(life_category);
```

**文件存储结构**：
```
ScreenTrace/
├── data/
│   ├── screenshots/
│   │   ├── 2026-03-05/
│   │   │   ├── 10-05-23.png

│   └── screenTrace.db

│   └── settings.json
```

**数据清理策略**：

- 同时删除对应的截图文件

## 数据流设计

### 完整数据流程

```
1. 触发阶段
   定时器/窗口切换 → 检查黑名单 → 通过 → 执行截图
                                    ↓
                               未通过 → 记录日志 → 结束

2. 处理阶段
   截图保存 → 相似度检测 →
                  ↓                      ↓
            相似度高(跳过)         相似度低(继续)
                  ↓                      ↓
               结束              图像压缩 → API调用
                                           ↓
                                    解析JSON结果
                                           ↓
                                    存入数据库 + 关联截图

3. 报告生成阶段
   用户请求 → 查询数据库 → 时间范围筛选 →
                               ↓
                    ┌──────────┼──────────┐
                    ↓          ↓          ↓
              时间线日志   任务归类统计   叙述式报告
                    ↓          ↓          ↓
                    └──────────┼──────────┘
                               ↓
                         可视化图表生成
                               ↓
                          输出报告文件
```

### 关键数据节点

**截图文件命名规则**：
- 格式：`YYYY-MM-DD/HH-MM-SS.png`
- 示例：`2026-03-05/10-05-23.png`

**数据库记录字段**：
- 时间戳（timestamp）
- 应用名（app_name）
- 生活维度（life_category）
- 活动形式（activity_form）
- 详细描述（description）
- 关键词（keywords）
- 截图路径（screenshot_path）
- 相似度分数（similarity_score）

## 错误处理与容错

### 关键错误场景

**1. API调用失败**
- **场景**：网络中断、API限流、余额不足
- **处理**：
  - 指数退避重试（1s → 2s → 4s → 最多3次）
  - 降级策略：跳过AI分析，仅保存截图和基础元数据
  - 写入错误日志，不弹窗打扰用户

**2. 截图权限问题**
- **场景**：某些应用禁止截图（DRM内容、管理员权限应用）
- **处理**：
  - 捕获异常，记录日志
  - 窗口标题标记为"[无法截图]"
  - 继续监控下一个窗口

**3. 磁盘空间不足**
- **场景**：截图过多导致磁盘满
- **处理**：
  - 启动时检查剩余空间
  - 空间<5GB时触发紧急清理
  - 配置最大存储空间上限

**4. 数据库损坏**
- **场景**：异常关机、硬件故障
- **处理**：
  - 定期自动备份（每周一次）
  - 启动时校验数据库完整性
  - 提供修复工具

**5. 黑名单误判**
- **场景**：关键词匹配导致正常工作被跳过
- **处理**：
  - 记录所有跳过事件到日志
  - 提供界面查看和恢复误判截图
  - 用户可调整黑名单规则

## 技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| API调用成本过高 | 经济负担 | 图像压缩、智能去重、缓存机制 |
| 黑名单误判率高 | 数据丢失 | 误判日志、手动恢复、规则优化 |
| 数据库性能下降 | 查询缓慢 | 索引优化、定期清理、分表策略 |
| 截图被其他应用拦截 | 数据缺失 | 降级处理、日志记录、用户提示 |
| 长期运行内存泄漏 | 系统卡顿 | 定期重启、内存监控、代码审查 |
